 1. Framework Package (Reusable library - Reusable framework - Reusable Across Projects - all core, shared)

 - It is a reusable framework package that should be installed via pip or Poetry and imported by the teams’ actual
 test projects.
 - Your “framework EcommerceAPI” must contain only reusable items!!
- Reusable by all teams
- Contains:
     - clients/utilities
     - helpers
     - DAOs
     - schemas (if generic)
     - config loaders
     - logging utilities
     - pytest plugins (optional: these can be packaged!)


Best-practice microservice testing structure for large orgs.
One shared framework repo:
qa-framework/
    EcommerceAPI/
        configs/
        data/
        schemas/
        src/
        pytest_plugins/
            logging.py
            cleanup.py
            reporting.py
        pyproject.toml
Note: Framework plugins should be in a package, not inside tests/.


It DOES NOT contain tests of any microservice.

3. How Plugins Work:

    Plugins for EACH PROJECT live in that project’s tests/plugins/ (import path:tests.plugins.logging).
    Each project’s conftest.py at project root declares:
    Python

pytest_plugins = [
    "tests.plugins.logging",
    "tests.plugins.reporting",
    "tests.plugins.entities",
    "tests.plugins.cleanup",
    "tests.plugins.api_fixtures"
]

If you want to share plugins between multiple projects, you can move those plugins into a plugins/ subpackage in the
framework (EcommerceAPI/src/plugins etc.) and import as EcommerceAPI.src.plugins.logging in any project.

🌟 How to Decide Where Helpers Go
If a helper is generic, reusable, and could be used by other microservices in the future:
    - Put it in your shared framework, EcommerceAPItest.
    - Examples: custom logging, DB utilities, Schema validation, authentication, context management, generic cleanup
    utilities.

If a helper is specific to ONE microservice (domain logic, API quirks, custom endpoints, etc.):
    - Keep it in your per-project test suite (e.g. TestEcommerceAPI/tests/helpers/).
    - Examples: customer_helper, orders_helper, coupons_helper if they only map to your ecommerce service’s
    endpoints/payloads.

As your framework matures:
    - Move truly reusable helpers/utilities up into EcommerceAPItest so other teams/projects can benefit without copy-paste.
    - Keep “business-specific” helpers in your microservice test repo.

🏗️ Example: Evolving Helpers
Suppose you have:
Python
# EcommerceAPItest/src/utilities/genericUtility.py    <- kept in framework

    ...
# TestEcommerceAPI/tests/helpers/customers_helper.py   <- kept in project suite
def create_customer(api_client, email, password):
    ...
If another team starts needing create_customer, refactor just enough for genericity, then move it up!
